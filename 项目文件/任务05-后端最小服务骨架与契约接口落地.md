# 任务05-后端最小服务骨架与契约接口落地

## 1. 任务目标
在同仓库前后端模式下，完成 backend 最小可用服务，要求：
1. 与 `contracts` 目录约定保持一致。
2. 提供可运行接口用于联调。
3. 具备基础错误返回与 traceId 能力。

## 2. 任务范围
- 创建 Node.js backend 服务骨架
- 实现 2 个核心接口：
- `GET /v1/config/schools`
- `POST /v1/import/logs`
- 实现健康检查接口：`GET /healthz`
- 添加运行与依赖配置
- 完成本地接口验收

## 3. 本次改动文件清单与作用说明
### 3.1 运行配置
1. `backend/package.json`
- 定义后端包信息与脚本：`start`、`dev`
- 引入依赖：`express`、`cors`

### 3.2 应用入口与装配
2. `backend/src/server.js`
- 读取端口并启动 HTTP 服务。

3. `backend/src/app.js`
- 组装 Express 应用：
- JSON 解析
- CORS
- traceId 中间件
- 路由挂载
- 404 统一错误返回

### 3.3 路由层
4. `backend/src/routes/configRoutes.js`
- 实现学校配置查询接口。

5. `backend/src/routes/importRoutes.js`
- 实现导入日志上报接口。
- 包含 `isValidImportLog(payload)` 基础字段校验。

### 3.4 服务层
6. `backend/src/services/configService.js`
- 从配置文件读取学校配置列表。

7. `backend/src/services/importLogService.js`
- 将导入日志按 NDJSON 追加写入 `runtime/import-logs.ndjson`。

### 3.5 配置样例与运行目录
8. `backend/src/config/schools.sample.json`
- 学校配置样例数据，用于接口返回验证。

9. `backend/runtime/`
- 运行时日志目录，避免把运行数据写入源码目录。

## 4. 核心函数说明
### 4.1 `createApp()`（`backend/src/app.js`）
- 输入：无
- 输出：Express app 实例
- 关键逻辑：
1. 注册中间件（CORS、JSON）
2. 生成并注入 `traceId`
3. 挂载配置路由与日志路由
4. 提供 `/healthz`
5. 统一 404 错误结构

### 4.2 `createConfigRoutes()`（`backend/src/routes/configRoutes.js`）
- 输入：无
- 输出：Router
- 关键逻辑：
1. 调用 `getSchoolConfigList()`
2. 成功返回 200
3. 失败返回 500 + 标准错误结构

### 4.3 `isValidImportLog(payload)`（`backend/src/routes/importRoutes.js`）
- 输入：日志 payload 对象
- 输出：布尔值
- 关键逻辑：
- 校验 `traceId/schoolId/errorCode/message/appVersion/platform/occurredAt` 必填且为非空字符串

### 4.4 `createImportRoutes()`（`backend/src/routes/importRoutes.js`）
- 输入：无
- 输出：Router
- 关键逻辑：
1. 校验 payload
2. 非法返回 400
3. 合法写入日志
4. 返回 202 + `accepted` + `traceId`

### 4.5 `getSchoolConfigList()`（`backend/src/services/configService.js`）
- 输入：无
- 输出：学校配置对象
- 关键逻辑：读取 JSON 文件并解析。

### 4.6 `appendImportLog(payload)`（`backend/src/services/importLogService.js`）
- 输入：日志 payload
- 输出：无
- 关键逻辑：按 NDJSON 追加写入，方便后续流式处理。

## 5. 架构设计说明
### 5.1 分层
- 路由层：HTTP 协议和返回码控制
- 服务层：业务与 IO 逻辑
- 配置层：数据源（样例配置）

### 5.2 可维护性设计
1. 接口与服务解耦，后续可替换配置来源（文件 -> 数据库）。
2. 统一错误结构，便于前端一致处理。
3. traceId 全链路注入，便于排障。

### 5.3 与前端分离关系
- 前端只依赖 `contracts` 契约，不依赖 backend 内部实现。
- backend 可以独立部署和演进。

## 6. 执行方案与关键决策
1. 决策：先用文件配置 + NDJSON 日志。
- 原因：最小可运行、便于快速联调。

2. 决策：基础校验放在路由层。
- 原因：先保证接口输入可控，后续可引入 schema validator。

3. 决策：先提供健康检查。
- 原因：便于部署探针与本地验收。

## 7. 验收标准
1. 依赖安装成功：`npm install` 无报错。
2. 服务可启动：监听 `http://localhost:3100`。
3. 接口验收通过：
- `GET /healthz` 返回 `status=ok`
- `GET /v1/config/schools` 返回样例配置
- `POST /v1/import/logs` 返回 `accepted=true`
4. 错误结构统一：失败响应包含 `traceId/errorCode/message`。

3. 返回结果符合预期：
