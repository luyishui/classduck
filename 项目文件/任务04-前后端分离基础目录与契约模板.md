# 任务04-前后端分离基础目录与契约模板

## 1. 任务目标
按你指定优先级，先完成前后端分离的基础工程能力建设：
1. 建立 backend/contracts/deploy 三层目录。
2. 建立可执行的 API 契约样板（OpenAPI + JSON Schema）。
3. 固化契约变更记录机制。

## 2. 任务范围
- 目录创建：`backend/`、`contracts/`、`deploy/`
- 文档创建：三目录 README
- 契约创建：OpenAPI v1 文件
- Schema 创建：配置列表、导入日志、错误响应
- Changelog 创建：契约变更记录模板

## 3. 本次改动文件清单与作用说明
### 3.1 后端工作区
1. `backend/README.md`
- 定义后端目录职责与服务边界。
- 指定后端服务模块：配置服务、日志服务、发布服务。

### 3.2 契约工作区
2. `contracts/README.md`
- 定义契约优先策略（contract-first）。

6. `contracts/schemas/error-response.schema.json`
- 标准错误响应结构。

### 3.3 部署工作区

## 4. 关键函数/结构说明
本任务主要是契约与目录建设，无业务函数实现，核心为接口结构设计：

### 4.1 `GET /v1/config/schools`
- 输入：无
- 输出：`SchoolConfigListResponse`
- 作用：客户端拉取可用学校导入配置。

### 4.2 `POST /v1/import/logs`
- 输入：`ImportLogRequest`
- 输出：`{ accepted, traceId }`
- 作用：客户端上报导入失败日志，支持后端追踪与统计。

### 4.3 错误结构 `ErrorResponse`
- 输入场景：400/500
- 输出字段：`traceId`、`errorCode`、`message`
- 作用：保证前后端异常处理协议统一。

## 5. 架构设计说明
### 5.1 分离边界
- 前端：`classduck_app/`，只消费契约，不依赖后端实现。
- 后端：`backend/`，可独立技术栈实现。
- 契约：`contracts/`，作为双方共同事实来源（single source of truth）。

### 5.2 依赖关系
- backend -> contracts
- frontend -> contracts
- deploy -> backend（部署流程）

## 6. 执行方案与关键决策
1. 决策：先建契约再建服务实现。
- 理由：避免接口反复改动，降低联调成本。

2. 决策：先定义最小必要接口。
- 理由：聚焦导入链路关键路径，后续再扩展功能接口。

3. 决策：统一错误响应结构。
- 理由：便于客户端统一重试、提示和日志追踪。
