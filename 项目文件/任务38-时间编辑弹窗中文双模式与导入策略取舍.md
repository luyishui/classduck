# 任务38：时间编辑弹窗中文双模式与导入策略取舍

## 1. 本轮目标
1. 修复课表节次时间编辑弹窗出现英文的问题。
2. 按需求改为“默认手动输入，右下按钮切换滚轮模式”。
3. 明确适配规则部署策略：本地私有 vs 云端下发的可行方案与推荐路径。

## 2. 代码实现
### 2.1 时间编辑弹窗重构
- 文件：`classduck_app/lib/features/schedule/ui/schedule_page.dart`
- 变更：
1. 将 `showTimePicker` 替换为 `_openTimeEditDialog()` 自定义弹窗。
2. 弹窗默认展示“手动输入模式”（小时/分钟输入框，中文文案）。
3. 右下按钮切换“滚轮模式”（小时轮 + 分钟轮）。
4. 支持在两种模式之间来回切换。
5. 增加输入合法性校验：小时 0-23，分钟 0-59。

### 2.2 已延续的相关优化
1. 一键批量对齐已改为上午/下午/晚上“分段内部对齐”。
2. 侧栏标题已改为“时间设置”。
3. 新建待办前会实时刷新关联课程，并行拉取课程列表。

## 3. 适配规则部署取舍（私有场景）

### 3.1 方案A：规则写死在 App 内（离线本地）
- 优点：
1. 完全不依赖服务器；
2. 私有部署最简单，外网不可见。
- 缺点：
1. 每次新增学校或修规则都要发新版 App；
2. 故障修复响应慢。

### 3.2 方案B：规则放私有后端接口（当前项目已有基础）
- 优点：
1. 不发版即可更新规则；
2. 可做灰度与回滚；
3. 多端统一规则。
- 缺点：
1. 需要维护后端可用性；
2. 需要鉴权与日志审计。

### 3.3 推荐（结合你“自跑、不公开”的诉求）
1. 主用方案B：规则放你自己的私有后端（内网或白名单访问）。
2. 同时保留本地兜底：App 内置一份最小规则，当后端不可用时不至于完全不可导入。
3. 私有化保障：
   - API 增加 token 或设备签名鉴权；
   - 规则文件不开放公共 CDN；
   - 关键脚本可放私有对象存储并设短时签名 URL。

## 4. 为什么你会看到“英文弹窗+切换不受控”
1. 之前直接调用系统 `showTimePicker`，不同平台/语言包下会出现英文元素。
2. 系统弹窗自带输入/旋钮切换按钮，但样式和行为由系统控制，无法严格按产品指定流程。
3. 自定义弹窗后，流程和文案都由我们掌控。
