# 任务37：时间设置分段对齐与导入方案白话说明

## 1. 本轮目标
1. 修正“时间设置 -> 一键批量对齐”逻辑：改为上午/下午/晚上三个时段分别内部对齐。
2. 将侧边栏分组标题“上课时间”改为“时间设置”。
3. 优化“新建待办 -> 关联课程”刷新体验，降低新增课程后显示延迟。
4. 输出“我们准备怎么做导入”的白话方案说明（不直接开做导入）。

## 2. 代码修改
### 2.1 课程表侧边栏
- 文件：`classduck_app/lib/features/schedule/ui/schedule_page.dart`
- 修改点：
1. 分组标题：`上课时间` -> `时间设置`。
2. `_ScheduleConfig.alignByTemplate()`：
   - 旧逻辑：从第1节开始一路顺排到最后一节（全局连续）。
   - 新逻辑：
     - 先对齐上午区间；
     - 再单独对齐下午区间；
     - 最后单独对齐晚上区间；
     - 时段之间不互相挤压，保留中间休息断层。

### 2.2 待办新建页课程联动
- 文件：`classduck_app/lib/features/todo/ui/todo_page.dart`
- 修改点：
1. `_openNewTodo()` 在跳转前先执行 `_loadCourseOptions()`，实时刷新课程选项。
2. `_loadCourseOptions()` 改为并行读取各课表课程，减少等待时间。

## 3. 本轮问题根因
### 3.1 一键批量对齐为什么错位
- 根因：之前把全天节次视为一条连续时间线，导致下午起始时间被上午尾部“带着走”。
- 结果：丢失午休/晚休这种“时段断层”。

### 3.2 关联课程为什么有时慢半拍
- 根因：课程选项主要在待办页加载时读取，跨页面新增课程后，缓存列表可能没马上刷新。
- 结果：用户进入新建待办的瞬间，看到的是上一轮课程集合。

## 4. 导入方案（白话）
> 先讲“怎么做”，本轮不动导入功能代码。

### 4.1 现在已有的基础能力
1. 后端有“规则文件”：`backend/src/config/adapter-rules.sample.json`。
2. 后端有“读取与下发接口”：
   - `backend/src/services/adapterRuleService.js`
   - `backend/src/routes/adapterRuleRoutes.js`
3. 前端有“拉规则并解析模型”：
   - `classduck_app/lib/features/import/data/adapter_rule_repository.dart`

### 4.2 我们准备怎么导入（建议分 6 步）
1. 第一步：拿到目标学校网站地址（用户正在登录的页面 URL）。
2. 第二步：用 URL 去匹配规则（host/path 命中哪条 `adapterId`）。
3. 第三步：按规则拿到解析脚本地址 `scriptUrl`，并检查 `needsLogin` 与 `delaySeconds`。
4. 第四步：在 WebView/页面上下文里执行脚本，拿到标准化课程 JSON。
5. 第五步：把 JSON 映射为我们的 `course_table` + `course` 数据结构，写入本地库。
6. 第六步：导入完成后触发课表页与待办课程选项刷新，保证“刚导入就可见”。

### 4.3 对照参考项目可读结论
1. `aishedule-master` 的思路是“三段式”：
   - `provider`：先把网页课表 HTML 抓出来；
   - `parser`：再把 HTML 解析成课程对象；
   - `timer`：单独计算每节课时间模板。
2. `Class-Shedule-Flutter` 的思路是“学校配置 + JS 脚本 + 标准 JSON 回传”：
   - 学校列表里配 URL 和脚本；
   - 脚本返回统一课程 JSON；
   - Flutter 端再落库。
3. 我们会吸收两者优点：
   - 规则可配置（像学校列表/规则表）；
   - 解析脚本可替换（像 parser 脚本）；
   - 时间模板可独立（像 timer）。
