# 任务09-导入执行页与日志上报链路

## 1. 任务目标
把导入模块从“学校列表展示”提升到“可执行流程占位 + 错误日志上报”：
1. 新增导入执行页面。
2. 建立前端日志上报仓储。
3. 打通前端 -> backend `/v1/import/logs` 链路。

## 2. 任务范围
- 扩展 HTTP 客户端 POST 能力
- 新增导入日志上报仓储
- 新增导入执行页面
- 学校列表点击跳转导入执行页
- 保留长按配置预览
- 前后端链路联调验收

## 4. 核心函数说明
### 4.1 `HttpJsonClient.postJsonMap(path, body)`
- 输入：接口路径、JSON 请求体
- 输出：JSON 对象响应
- 逻辑：
1. POST 请求
2. 校验 2xx
3. 解析 JSON
4. 异常统一抛 `ApiException`

### 4.2 `ImportLogRepository.reportImportFailure(...)`
- 输入：schoolId、errorCode、message、appVersion、platform
- 输出：服务端 traceId
- 逻辑：
1. 生成客户端 traceId
2. 组装日志 payload
3. 调用 `/v1/import/logs`
4. 返回服务端 traceId

### 4.3 `ImportExecutionPage._reportFailureSample()`
- 输入：用户点击事件
- 输出：UI 状态更新
- 逻辑：
1. 进入提交中状态
2. 调用日志仓储上报模拟错误
3. 成功展示 traceId
4. 失败展示错误信息

### 4.4 `ImportSchoolListPage._openExecutionPage(config)`
- 输入：学校配置对象
- 输出：导航行为
- 逻辑：跳转导入执行页面并传递配置。

## 5. 架构设计说明
1. 导入 UI 与日志上报逻辑分离：
- UI 页面只调仓储，不直接写 HTTP。

2. 链路可观测性提升：
- 每次日志上报都返回 traceId，便于后端追踪。

3. 行为分层清晰：
- 学校列表页负责选择来源。
- 执行页负责执行动作与上报。

## 6. 执行方案与关键决策
1. 决策：先实现“模拟失败上报”。
- 原因：在 WebView 与解析脚本尚未接入前，先验证关键异常链路。

2. 决策：点击进入执行页、长按看配置。
- 原因：兼顾主流程效率与调试可见性。

3. 决策：复用统一 HTTP 客户端。
- 原因：避免 GET/POST 逻辑分散，便于统一错误处理。

## 7. 验收标准
1. `flutter analyze` 无静态错误。
2. 学校列表点击可进入执行页。
3. 执行页点击“Simulate Parse Failure Log Upload”会触发上报。
4. backend `/v1/import/logs` 返回 `accepted=true` 且包含 traceId。
