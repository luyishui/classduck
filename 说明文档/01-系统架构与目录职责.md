# 系统架构与目录职责

## 1. 顶层目录
1. `classduck_app/`
- Flutter 客户端主工程（iOS/Android 目标）。

2. `backend/`
- 后端能力工程（配置/策略/日志接口），用于前后端分离。

3. `contracts/`
- 前后端契约定义目录（请求响应结构、错误码、版本规范）。

4. `deploy/`
- 部署模板与环境脚本。

5. `参考文件/`
- 仅参考输入（PRD、Pen、历史项目），不作为运行时依赖。

## 2. Flutter 工程分层
1. `lib/app/`
- 应用壳、路由、全局主题与初始化。

2. `lib/shared/`
- 全局设计 token、复用组件（弹窗、基础卡片）、公共工具。

3. `lib/features/schedule/`
- 课表域：页面、交互、数据读取、导入联动。

4. `lib/features/todo/`
- 待办域：分栏、分组抽屉、新建流程、状态迁移。

5. `lib/features/profile/`
- 我的域：统计卡、设置入口、统计弹窗、今日状态弹窗。

6. `lib/features/import/`
- 导入域：学校选择、导入执行、冲突决策。
- 适配规则读取：通过 `adapter_rule_repository.dart` 获取后端解析规则。

7. `lib/features/settings/`
- 外观与主题、提醒与通知、关于页与版本更新。

8. `lib/data/local/`
- SQLite 本地数据访问层（表结构、DAO/Repository 底座）。

## 3. 依赖方向规范
1. 页面（ui）只调用应用/仓储接口，不直接操作底层数据库连接。
2. Repository 负责数据源聚合与持久化边界。
3. shared 层不依赖 feature 业务层。

## 4. 维护约定
1. 页面复杂交互必须抽出私有函数并加注释。
2. 任何非直观业务判断（如“已上课程”判定）必须注释清楚。
3. UI 以 PRD + Pen 双对齐；允许在不破坏语义前提下做可用性增强。
